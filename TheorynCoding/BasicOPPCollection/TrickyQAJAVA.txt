### 1) HashMap — Null Keys

**Q:** How does `HashMap` handle `null` keys?

```java
Map<String,String> map=new HashMap<>();
map.put(null,"A");
map.put(null,"B");
System.out.println(map.get(null)); // B
```

**A:** Only one `null` key is allowed; new put replaces old.
**Difficulty:** Basic

---

### 2) HashMap — Key Mutation

**Q:** Why mutable keys are dangerous in `HashMap`?

```java
class Key{int id;Key(int id){this.id=id;}
 public int hashCode(){return id;}
 public boolean equals(Object o){return ((Key)o).id==id;}}
Map<Key,String> map=new HashMap<>();
Key k=new Key(1);map.put(k,"X");
k.id=2;System.out.println(map.get(k)); // null ❌
```

**A:** Mutating key changes hash → entry lost.
**Difficulty:** Intermediate

### Batch — SOLID (30 Q\&A).

Each entry: **Q**, small **code** (if helpful), **A** (concise), **Difficulty**.

---

### 1) Single Responsibility Principle (SRP) — definition

**Q:** What is SRP?
**A:** A class should have exactly one reason to change — one responsibility. Keeps code modular and easier to test.
**Difficulty:** Basic

---

### 2) SRP — violation example

**Q:** Show a simple SRP violation.

```java
class OrderService {
  void placeOrder(Order o){ /* validate + persist + notify */ }
}
```

**A:** `OrderService` handles validation, persistence, and notification — multiple reasons to change. Split responsibilities (validator, repository, notifier).
**Difficulty:** Basic

---

### 3) SRP — refactor to follow SRP

**Q:** Quick refactor of previous violation.

```java
class OrderValidator { boolean valid(Order o){...} }
class OrderRepository { void save(Order o){...} }
class OrderNotifier { void notify(Order o){...} }
class OrderService {
  OrderValidator v; OrderRepository r; OrderNotifier n;
  void placeOrder(Order o){ if(v.valid(o)){ r.save(o); n.notify(o); } }
}
```

**A:** Each class has single responsibility — changes in notification don’t affect persistence.
**Difficulty:** Intermediate

---

### 4) Open/Closed Principle (OCP) — definition

**Q:** What is OCP?
**A:** Software entities should be open for extension but closed for modification — add behavior by extending, not changing existing code.
**Difficulty:** Basic

---

### 5) OCP — example with strategy pattern

**Q:** How to extend behavior without modifying code?

```java
interface Discount { double apply(double price); }
class NoDiscount implements Discount { public double apply(double p){return p;} }
class BlackFriday implements Discount { public double apply(double p){return p*0.7;} }
class Checkout { Discount d; double total(double p){ return d.apply(p); } }
```

**A:** Add new `Discount` implementations without changing `Checkout` — follows OCP.
**Difficulty:** Intermediate

---

### 6) OCP — pitfalls (when overused)

**Q:** Can OCP lead to excessive classes?
**A:** Yes — blind extension can bloat code. Favor pragmatic designs: use configuration or composition where appropriate.
**Difficulty:** Tricky

---

### 7) Liskov Substitution Principle (LSP) — definition

**Q:** What is LSP?
**A:** Subtypes must be substitutable for their base types without altering desirable program properties — derived classes should not strengthen preconditions or weaken postconditions.
**Difficulty:** Basic

---

### 8) LSP — classic Rectangle/Square pitfall

**Q:** Why `Square extends Rectangle` often violates LSP?

```java
class Rectangle { void setWidth(int w); void setHeight(int h); }
class Square extends Rectangle { void setWidth(int w){ super.setWidth(w); super.setHeight(w);} }
```

**A:** `Square` changes behaviour of `setWidth`/`setHeight` — client expecting independent width/height breaks. Use composition instead.
**Difficulty:** Tricky

---

### 9) LSP — correct approach for rectangle/square

**Q:** How to model correctly?

```java
interface Shape { int area(); }
class Rectangle implements Shape { ... }
class Square implements Shape { ... } // separate implementations
```

**A:** Treat square and rectangle as separate implementations of `Shape` instead of subclassing.
**Difficulty:** Intermediate

---

### 10) Interface Segregation Principle (ISP) — definition

**Q:** What is ISP?
**A:** Clients should not be forced to depend on interfaces they don’t use — prefer many small, client-specific interfaces over one large interface.
**Difficulty:** Basic

---

### 11) ISP — example (fat interface)

**Q:** Show fat interface problem.

```java
interface Worker { void work(); void eat(); }
class Robot implements Worker { public void eat(){ throw new UnsupportedOperationException(); } }
```

**A:** `Robot` forced to implement `eat()` it can't do — violates ISP. Split into `Workable` and `Eatable`.
**Difficulty:** Intermediate

---

### 12) ISP — refactor example

**Q:** Refactor the fat interface.

```java
interface Workable{ void work(); }
interface Eatable{ void eat(); }
class Human implements Workable, Eatable{...}
class Robot implements Workable{...}
```

**A:** Clients implement only what they need.
**Difficulty:** Basic

---

### 13) Dependency Inversion Principle (DIP) — definition

**Q:** What is DIP?
**A:** High-level modules should not depend on low-level modules; both should depend on abstractions. Abstractions should not depend on details; details depend on abstractions.
**Difficulty:** Basic

---

### 14) DIP — example with constructor injection

**Q:** How to apply DIP in code?

```java
interface Payment { void pay(); }
class CreditCard implements Payment{ public void pay(){...} }
class Checkout {
  private final Payment payment;
  Checkout(Payment p){ this.payment = p; }
  void doPay(){ payment.pay(); }
}
```

**A:** `Checkout` depends on `Payment` abstraction, not concrete `CreditCard` — DIP applied.
**Difficulty:** Intermediate

---

### 15) DIP — factory vs DI container

**Q:** Is using factory violating DIP?
**A:** Not necessarily — factories still return abstractions. DI containers often implement DIP by wiring abstractions to implementations at runtime. Key is that high-level code depends on interfaces, not concrete classes.
**Difficulty:** Intermediate

---

### 16) SRP vs DIP conflict example

**Q:** Can SRP and DIP conflict?
**A:** Yes — introducing an interface per class to satisfy DIP may increase number of files; balance is needed. SRP/DIP should be applied pragmatically, not dogmatically.
**Difficulty:** Tricky

---

### 17) SOLID and testing

**Q:** How SOLID helps unit testing?
**A:** Small, single-responsibility classes and dependency inversion (inject mocks) make unit tests focused and easy to write.
**Difficulty:** Basic

---

### 18) Interface Segregation — client-specific interfaces (example)

**Q:** Provide an ISP example with printers.

```java

interface ScanJob { void scan(); }
class MultiFunctionPrinter implements PrintJob, ScanJob{...}
class SimplePrinter implements PrintJob{...}
```

**A:** Clients implement only needed interfaces; no unsupported methods.
**Difficulty:** Basic

---

### 19) DIP — field injection vs constructor injection

**Q:** Which injection is preferable for DIP and why?
**A:** Constructor injection is preferred — it makes dependencies explicit, supports immutability, and aids testability. Field injection hides dependencies and complicates testing.
**Difficulty:** Intermediate

---

### 20) OCP applied to validation rules

**Q:** How to add new validation rules without changing existing validator?

```java

  List<Rule> rules;
  boolean valid(Order o){ return rules.stream().allMatch(r->r.validate(o)); }
}
```

**A:** Add new `Rule` implementations and register them — no change to `CompositeValidator`.
**Difficulty:** Intermediate

---

### 21) LSP — behavioral contract example (pre/postconditions)

**Q:** How can a subtype violate LSP with stronger precondition?
**A:** If base method accepts `null` but subtype rejects it (throws NPE), clients expecting null-tolerant behavior break — subtype must honor base contract.
**Difficulty:** Tricky

---

### 22) Interface Segregation — ISP and APIs

**Q:** How ISP applies to public APIs?
**A:** Expose small focused interfaces for different client groups (read vs write interfaces), reducing coupling and evolution impact.
**Difficulty:** Intermediate

---

### 23) DIP — inversion via events (alternative)

**Q:** How to invert dependency using events?

**A:** High-level module emits events, low-level modules subscribe — decouples modules and follows DIP.

### 24) SOLID in microservices context

**Q:** How SOLID maps to microservices?
**A:** SRP → single responsibility per service; OCP → services extensible via contracts; DIP → services depend on abstractions (APIs), not concrete implementations; focus on loosely coupled, replaceable services.
**Difficulty:** Intermediate

---

### 25) SOLID trade-offs — over-abstraction risk

**Q:** What is a risk of strictly applying SOLID?
**A:** Over-abstraction, too many interfaces and indirection, increased complexity and harder to navigate. Apply SOLID where it delivers maintainability benefits.
**Difficulty:** Tricky

---

### 26) LSP — co/contra-variance in Java (methods)

**Q:** Can overridden methods change return type?
**A:** Yes — covariant return types are allowed (return subtype). Parameters cannot be weakened/strengthened arbitrarily — changing parameter types breaks substitution.
**Difficulty:** Intermediate

---

### 27) DIP — example with adapter pattern

**Q:** How adapter helps DIP?

```java

  LegacyPayment legacy;
  public void pay(){ legacy.doPay(); }
}
```

**A:** Adapter implements the abstraction expected by high-level module, allowing low-level legacy to fit DIP.
**Difficulty:** Intermediate

### 28) SRP — single responsibility vs cohesive grouping

**Q:** How to choose responsibility granularity?
**A:** Responsibility should align with cohesive behavior: group closely related functions; avoid ultra-fine splitting that harms readability.
**Difficulty:** Tricky

---

### 29) SOLID & functional style

**Q:** Do SOLID principles apply to functional programming?
**A:** Yes — concerns translate: single responsibility (pure functions), interface segregation (small function contracts), dependency inversion (pass functions as parameters), and OCP/LSP become function composition and preservation of behavior.
**Difficulty:** Intermediate

### 30) Practical checklist to evaluate SOLID compliance

**Q:** Quick checklist to judge SOLID for a class/module?
**A:**

* SRP: One reason to change?
* OCP: Add behavior without modifying existing code?
* LSP: Can subclass replace parent safely?
* ISP: Are interfaces client-specific and minimal?
* DIP: Do high-level modules depend on abstractions?

### Batch — Multithreading & Concurrency (30 Q\&A)

### 1) Thread vs Runnable vs Callable

**Q:** When use `Thread`, `Runnable`, or `Callable`?

```java
Runnable r = () -> doWork();
Callable<Integer> c = () -> 42;
```

**A:** `Runnable` — task with no result, `Callable` — returns result/throws checked exceptions, use with `ExecutorService`. Prefer `Runnable/Callable` over extending `Thread`.
**Difficulty:** Basic

### 2) Starting a thread — correct way

**Q:** What is wrong with calling `run()` directly?

```java
Thread t = new Thread(() -> System.out.println("hi"));
t.run(); // runs on current thread
t.start(); // starts new thread
```

**A:** `run()` executes on current thread; `start()` creates a new OS/VM thread.
**Difficulty:** Basic

---

### 3) join() usage

**Q:** What does `join()` do?

```java
t.start(); t.join(); // current thread waits for t to finish
```

**A:** Causes current thread to wait until target thread terminates (optionally with timeout).
**Difficulty:** Basic

---

### 4) sleep() vs wait()

**Q:** Difference between `Thread.sleep()` and `Object.wait()`?
**A:** `sleep()` pauses without releasing locks; `wait()` releases monitor and needs notify/notifyAll to resume; `wait()` must be called inside synchronized block.
**Difficulty:** Intermediate

---

### 5) volatile — visibility not atomicity

**Q:** Can `volatile` make `++` atomic?

```java
volatile int x; x++; // still not atomic
```

**A:** `volatile` ensures visibility and ordering (no caching), but compound actions (read-modify-write) are not atomic — use `AtomicInteger` or synchronization.
**Difficulty:** Intermediate

---

### 6) happens-before — brief meaning

**Q:** What is the *happens-before* relationship?
**A:** Memory-consistency guarantee defining ordering/visibility: actions before a release (e.g., unlock/write volatile) are visible after acquire (lock/read volatile) — essential for correctness.
**Difficulty:** Tricky

---

### 7) synchronized — what it locks on

**Q:** What does `synchronized(this)` lock?
**A:** Locks the monitor of the given object. `synchronized(staticMethod)` locks the `Class` object for that class.
**Difficulty:** Basic

---

### 8) ReentrantLock vs synchronized

**Q:** Why use `ReentrantLock` over `synchronized`?
**A:** `ReentrantLock` offers tryLock, timed waits, fairness options, multiple Conditions; `synchronized` is simpler and less error-prone for basic use.
**Difficulty:** Intermediate

---

### 9) wait/notify vs Condition

**Q:** Advantage of `Condition` over `wait/notify`?
**A:** `Condition` provides multiple condition objects per lock and clearer await/signal semantics (like multiple queues), avoiding single-notify-all ambiguity.
**Difficulty:** Intermediate

---

### 10) Deadlock example & avoidance

**Q:** Give a simple deadlock and how to avoid it.

```java
synchronized(a){ synchronized(b){ /*...*/ } }
synchronized(b){ synchronized(a){ /*...*/ } } // deadlock
```

**A:** Deadlock = circular wait. Avoid by consistent lock ordering, using `tryLock` with timeout, or reducing lock scope.
**Difficulty:** Tricky

---

### 11) Livelock vs deadlock vs starvation

**Q:** Difference in one sentence.
**A:** Deadlock — threads blocked forever waiting locks; Livelock — threads active but make no progress (reacting to each other); Starvation — thread indefinitely denied CPU or resources.
**Difficulty:** Tricky

---

### 12) ThreadLocal purpose & leak risk

**Q:** Why use `ThreadLocal` and how can it leak?

```java
ThreadLocal<Formatter> tl = ThreadLocal.withInitial(() -> new Formatter());
tl.remove(); // avoid leak in thread pools
```

**A:** ThreadLocal gives thread-confined state. In thread pools, failing to `remove()` leaves values attached to long-lived threads causing memory leaks.
**Difficulty:** Intermediate

### 13) AtomicInteger CAS example

**Q:** How does `AtomicInteger` ensure atomic increment?

```java
AtomicInteger a = new AtomicInteger(0);
a.incrementAndGet(); // uses CAS
```

**A:** Uses low-level compare-and-swap (CAS) to loop and update without locks — lock-free atomic operations.
**Difficulty:** Intermediate

### 14) volatile + double-checked locking safe pattern

**Q:** How to implement lazy singleton safely?

```java
public static S get(){ if(instance==null){ synchronized(S.class){ if(instance==null) instance = new S(); } } return instance; }
```

**A:** `volatile` prevents reordering; double-checked locking with volatile yields lazy, thread-safe init.

# 2) Dependency Injection (DI) — patterns & small examples

## Constructor injection (recommended)

* Advantages: immutable dependencies, easier testing, explicit mandatory dependencies.

```java
@Component
public class OrderService {
  private final PaymentGateway gateway;
  public OrderService(PaymentGateway gateway) {
    this.gateway = gateway;
  }
}
```

Use `@Autowired` on constructor only if necessary (since Spring 4.3 single constructor auto-wires).

## Setter injection (optional)

* Use when dependency is optional or cyclic.

```java
@Component
public class A {
  private B b;
  @Autowired
  public void setB(B b){ this.b = b; }
}
```

## Field injection (discouraged in production code)

* Simple but hides dependencies and makes testing/hierarchy harder.

```java
@Component
class C {
  @Autowired private D d; // avoid if possible
}
```

## Qualifier & Primary

* When multiple beans of same type exist:

```java
@Component @Primary class FastPayment implements Payment {}
@Component @Qualifier("slow") class SlowPayment implements Payment {}
// Inject with @Qualifier("slow")
```

## @ConfigurationProperties (type-safe config DI)

```java
@ConfigurationProperties("app")
@Component
public class AppProps { private String name; /* getters/setters */ }
```

# 3) AOP — small example and gotchas

## Simple Aspect with `@Around`

```java
@Aspect
@Component
public class PerfAspect {
  @Around("@annotation(org.springframework.web.bind.annotation.GetMapping)")
  public Object time(ProceedingJoinPoint pjp) throws Throwable {
    long t = System.nanoTime();
    try { return pjp.proceed(); }
    finally { System.out.println("elapsed: " + (System.nanoTime()-t)); }
  }
}
```

* `@EnableAspectJAutoProxy` enables proxying (Spring Boot auto-enables AOP when aspect dependency present).
* Pointcuts can be by annotation, package, method name, etc.

## Common gotchas

* **Proxy-based AOP:** Spring uses proxies — by default JDK dynamic proxies for interfaces, else CGLIB for classes. If you need class proxying, use `@EnableAspectJAutoProxy(proxyTargetClass=true)`.
* **Self-invocation:** calling an advised method from the same class bypasses proxy → advice not applied. Solution: call via injected proxy or refactor to another bean.
* **Visibility:** Proxies usually advise public methods; private/final methods may not be advised depending on proxy type (CGLIB can advise non-final methods).
* **Order:** Use `@Order` or `Ordered` to control aspect ordering.

# 4) `@Transactional` specifics tied to AOP

* `@Transactional` is implemented using proxies; hence:

  * Must be on public methods (best practice) if you rely on proxying.
  * Self-invocation will bypass transactional behavior.
  * For class-based proxies, CGLIB proxies can be used (`proxyTargetClass=true`).
* Example:

```java
@Service
public class S {
  @Transactional
  public void save(){ ... }
  public void doAll(){ save(); } // transactional behavior bypassed on self-call
}
```

# 5) Quick Best Practices

* Prefer **constructor injection**. Avoid field injection in production classes.
* Know the difference **proxy vs target** (AOP) — watch for self-invocation and visibility.
* Use `@ConfigurationProperties` for structured config, `@Value` for quick single values.
* Use `@Repository` for DAOs (translation of exceptions).
* Keep aspects small and focused; avoid heavy work in advice; be explicit with pointcuts.
* For testing, use `@MockBean` and slice tests (`@WebMvcTest`, `@DataJpaTest`) to speed up tests.
* For beans which must be proxied (transactional/aspected), avoid `final` methods/classes.

# 🔹 Hibernate — Full Q\&A Pack (Grouped by Topics)

## 1. Basics (Foundations)

1. **What is Hibernate?**

   * Hibernate is an ORM (Object Relational Mapping) framework that maps Java classes to DB tables and handles persistence logic automatically.
   * Benefits: reduces boilerplate JDBC, handles transactions, caching, lazy loading, etc.

2. **Difference between Hibernate & JDBC?**

   * JDBC: manual SQL, result set mapping, transaction handling.
   * Hibernate: automatic mapping, HQL/Criteria queries, caching, declarative transaction management.

3. **What is ORM?**

   * ORM = Object Relational Mapping → mapping Java objects to relational DB tables using metadata/annotations/XML.

---

## 2. Session, SessionFactory & Transactions

4. **What is Session in Hibernate?**

   * Main interface between app & DB. Short-lived, not thread-safe, manages persistence context (L1 cache).

5. **What is SessionFactory?**

   * Heavy-weight, thread-safe object; used to create sessions. Usually one per DB.

6. **How to open/close a Session?**

   ```java
   SessionFactory sf = new Configuration().configure().buildSessionFactory();
   Session s = sf.openSession();
   s.beginTransaction();
   // work
   s.getTransaction().commit();
   s.close();
   ```

7. **Difference between openSession() and getCurrentSession()?**

   * `openSession()` → always creates new Session, must be closed manually.
   * `getCurrentSession()` → returns session bound to context (thread/transaction), auto-closed.

8. **How transactions are managed?**

   * Hibernate provides `Transaction` API (`beginTransaction()`, `commit()`, `rollback()`).
   * With Spring: `@Transactional` is used.

---

## 3. Caching

9. **What is 1st-level cache?**

   * Built-in session-scoped cache (mandatory).
   * Example: same entity in one session will not hit DB twice.

10. **What is 2nd-level cache?**

* Optional cache at SessionFactory level (EhCache, Hazelcast, Infinispan).
* Configured via annotations (`@Cacheable`, `@Cache`).

11. **What is Query cache?**

* Caches results of queries (depends on 2nd-level cache).

```java
Query q = s.createQuery("from User");
q.setCacheable(true);
```

12. **Explain cache concurrency strategies.**

* `READ_ONLY`, `NONSTRICT_READ_WRITE`, `READ_WRITE`, `TRANSACTIONAL`.

13. **What is NaturalId cache?**

* Caches entities by business key (e.g., email).

```java
@NaturalId @Column(unique=true) String email;
```

---

## 4. Mappings

14. **What annotations are used for mapping?**

* `@Entity`, `@Table`, `@Id`, `@GeneratedValue`, `@Column`, `@OneToOne`, `@OneToMany`, `@ManyToOne`, `@ManyToMany`.

15. **Difference between `@OneToMany` and `@ManyToOne`?**

* `@OneToMany`: one entity linked with multiple.
* `@ManyToOne`: many entities reference one.

16. **What is Cascade in Hibernate?**

* Defines operations propagated from parent to child (`CascadeType.PERSIST`, `REMOVE`, etc.).

17. **Difference between lazy & eager fetching?**

* Lazy = association loaded on demand (default).
* Eager = association loaded immediately with parent.

18. **What is orphanRemoval?**

* If true, removing child from parent’s collection deletes it from DB.

---

## 5. Queries

19. **What is HQL?**

* Hibernate Query Language → object-oriented, works with entity names instead of table names.

```java
Query q = session.createQuery("from User where name=:n");
```

20. **Difference between HQL and SQL?**

* HQL is object-centric (uses entity, property).
* SQL is DB-centric (uses tables, columns).

21. **What is Criteria API?**

* Type-safe, programmatic way to build queries.

```java
CriteriaBuilder cb = session.getCriteriaBuilder();
CriteriaQuery<User> cq = cb.createQuery(User.class);
Root<User> root = cq.from(User.class);
cq.select(root).where(cb.equal(root.get("name"), "John"));
```

22. **Named Queries?**

* Queries defined with annotations or XML.

```java
@NamedQuery(name="User.findByName", query="from User where name = :name")
```

23. **Native SQL in Hibernate?**

```java
SQLQuery q = session.createSQLQuery("select * from user");
```

## 6. States of Entity

24. **Entity states in Hibernate?**

* **Transient**: new object, not associated with session/DB.
* **Persistent**: associated with session, tracked.
* **Detached**: was persistent, session closed.
* **Removed**: marked for deletion.

25. **Difference between save(), persist(), merge(), update()?**

* `save()` → returns identifier, inserts immediately (sometimes before flush).
* `persist()` → follows JPA spec, doesn’t return ID.
* `merge()` → copies detached entity state into persistence context.
* `update()` → reattaches detached entity.

## 7. Transactions & Flush

26. **What is flush?**

* Synchronizes session state with DB (executes pending SQL).
* Triggered by commit, or manually via `session.flush()`.

27. **Flush modes?**

* `AUTO` (default), `COMMIT`, `ALWAYS`, `MANUAL`.

28. **What is dirty checking?**

* Hibernate detects changes to persistent objects and updates DB automatically on flush/commit.

## 8. Performance Tuning

29. **How to avoid N+1 select problem?**

* Use `JOIN FETCH` in HQL or `@Fetch(FetchMode.JOIN)`.

30. **Batch fetching?**

* Fetches collections in batches using `hibernate.default_batch_fetch_size`.

31. **LazyInitializationException?**

* Occurs when accessing lazy collection outside session.

## 9. Advanced Topics

32. **Inheritance mapping strategies?**

* `SINGLE_TABLE`, `JOINED`, `TABLE_PER_CLASS`.

33. **What is optimistic vs pessimistic locking?**

* Optimistic = versioning (`@Version`).
* Pessimistic = DB-level locks (`LockModeType.PESSIMISTIC_WRITE`).

34. **What is dirty checking optimization?**

* Hibernate uses snapshots to compare entity states and only updates changed fields.

35. **Integrating with Spring Boot?**

* Boot auto-configures Hibernate via `spring-boot-starter-data-jpa`.

## 10. Interview Tricky Qs

36. **Difference between `get()` and `load()`?**

## 🔹 A. Core Java — tricky language/runtime bits (21)

**0. Constructor Chaining — what is it?**
Calling one constructor from another to reuse initialization logic.

* `this(...)` → same class
* `super(...)` → parent class
* Must be **first statement**
* Cannot use both in same constructor
  👉 Ensures reuse + clean init order.

**1. Static initialization order**
Static fields/blocks → instance blocks → constructor.

**2. Instance initialization block vs constructor**
Instance initializer runs before constructor, each time.

**3. final vs finally vs finalize()**

* `final`: keyword
* `finally`: cleanup block
* `finalize()`: deprecated GC hook.

**4. Covariant return types**
Subclass method can return a subtype of overridden method.

**5. Variable shadowing vs method hiding**
Fields: shadowed; static methods: hidden, not overridden.

**6. Pass-by-value**
Java always passes by value; object references copied.

**7. Cloneable pitfalls**
`Object.clone()` shallow; deep copy requires manual code.

**8. Serialization quirks**
`transient` skips fields; `serialVersionUID` ensures version compatibility.

**9. String.intern()**
Adds literal to string pool; saves memory but can increase metaspace use.

**10. Integer caching**
Between –128 and 127, `==` works due to cache.

**11. Varargs ambiguity**
Overloading with varargs may cause resolution surprises.

**12. try-with-resources**
Suppressed exceptions retrievable with `getSuppressed()`.

**13. Checked vs unchecked exceptions**
Checked: recoverable, must handle. Unchecked: programming bugs.

**14. Method reference vs lambda `this`**
Lambda: `this` = outer class. Inner class: `this` = inner.

**15. Widening vs boxing**
Widening preferred over boxing in overload resolution.

**16. equals/hashCode contract**
Equal objects must have same hashCode; HashSet example.

**17. ClassLoader delegation model**
Parent-first delegation; custom loaders risk duplicates.

**18. Reflection setAccessible(true)**
Breaks encapsulation; slow; unsafe under SecurityManager.

**19. Default method diamond conflict**
Two interfaces with same default → must override or qualify.

**20. Records limitations**
Immutable, final, can’t extend classes; only implement interfaces.

## 🔹 B. OOP / SOLID / Patterns (10)

**21. Interface vs Abstract class**
Interface = contract; Abstract = base with state/behavior.

**22. Default methods pitfalls**
Adding default methods may break existing implementations.

**23. Singleton pitfalls**
Reflection & serialization break classic singleton → use enum.

**24. Factory Method vs Abstract Factory**
Factory Method → one product. Abstract Factory → family of products.

**25. LSP violation**
Subclass must not strengthen preconditions (Rectangle/Square problem).

**26. Strategy vs State vs Command**
Strategy = interchangeable algorithms; State = internal mode; Command = request as object.

**27. Visitor trade-off**
Easy to add operations; hard to add new elements.

**28. Law of Demeter**
Talk to friends, not strangers → reduce coupling.

**29. GRASP Controller/Creator**
Controller mediates events; Creator = best owner of created objects.

**30. Overengineering with patterns**
Using patterns unnecessarily makes design rigid.

## 🔹 C. Concurrency & Threading (12)

**31. start() vs run()**
`start()` spawns thread; `run()` runs in same thread.

**32. interrupt() behavior**
Sets flag; blocking ops throw InterruptedException.

**33. wait() vs sleep()**
wait() releases monitor; sleep() does not.

**34. volatile ≠ atomic**
Only visibility; not atomic. Use AtomicInteger or locks.

**35. Double-checked locking (pre-Java 5)**
Broken due to reordering; fix with volatile.

**36. ThreadLocal leaks**
In pools, values persist; always call `remove()`.

**37. Deadlock vs Livelock vs Starvation**
Deadlock = stuck; Livelock = retrying no progress; Starvation = low-priority never runs.

**38. join() + interrupt**
Interrupted thread in join → InterruptedException.

**39. CompletableFuture exception handling**
Use `handle`, `exceptionally`, `whenComplete`.

**40. Fair vs unfair locks**
Fair prevents starvation but reduces throughput.

**41. StampedLock optimistic reads**
Validate stamp before trusting value.

**42. ForkJoinPool blocking hazard**
Blocking tasks exhaust pool → use ManagedBlocker.

## 🔹 D. Spring (MVC, Boot, DI, AOP) (18)

**43. Bean lifecycle order**
Constructor → @PostConstruct → afterPropertiesSet → initMethod.

**44. Prototype in singleton**
Use scoped proxy for per-request beans.

**45. @Transactional pitfalls**
Self-invocation, non-public methods, checked exceptions not rolling back.

**46. JDK vs CGLIB proxy**
JDK = interface only; CGLIB = subclass; final/private not advised.

**47. @Bean vs @Component vs @Import**
@Bean = method; @Component = scanned; @Import = import config.

**48. Conditional annotations**

**49. Circular dependencies**

**50. OpenSessionInView anti-pattern**

**53. Aspect ordering**
Lower @Order runs outermost.

**54. BeanPostProcessor vs BeanFactoryPostProcessor**
BPP = beans; BFPP = definitions.

**55. @Configuration proxyBeanMethods**
Default true ensures singleton; set false if not needed.

**56. Security filter chain ordering**
Wrong order breaks CORS/auth.

**57. WebFlux blocking ops**

# How to present in interviews — exact phrasing & demo tips

* **Start with the rule** (one-liner). Example: “Constructor chaining is calling one constructor from another using `this(...)` or `super(...)` and it must be the first statement.”
* **Show 10–20 line code** (live or paste) and run it if allowed. Point out output and ordering.
* **Explain tradeoffs** briefly: why use chaining (DRY of init), what to avoid (circular calls).
* **Finish with an edge-case**: “Note you can’t call both `this` and `super` in same constructor; compiler enforces this.”

For concurrency, show a failing example first (race condition) then the fix — interviewers like to see you reason from broken→fixed.
